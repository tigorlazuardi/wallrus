# 006-service-images builder notes

## Invocation 1 observations

### FTS table name

Confirmed by reading `drizzle/migrations/0001_fts5_search_text.sql`: the virtual table is named
`images_fts` (NOT `search_text_fts` as the slice prose says in some places). The FTS table has one
content column: `search_text`. The `content_rowid='rowid'` means the FTS rowid maps to `images.rowid`,
NOT to `images.id` (which is a UUID text PK). This means the IN-subquery cannot use `images_fts.id` directly
-- the FTS table doesn't have an `id` column. The reconciliation workaround: join via rowid.

The correct IN-subquery is:
`AND images.rowid IN (SELECT rowid FROM images_fts WHERE images_fts MATCH ?)`

### bm25 direction

Per IMPLEMENTATION.md Gotchas: `bm25()` returns negative scores; ORDER BY bm25 ASC = best-first.
Must use `ORDER BY bm25(images_fts) ASC` (the least-negative = best match comes first).

### favorited derivation

`favorites` table shape: `{ image_id PK, favorited_at }`. The DTO `favorited` field is derived via
a LEFT JOIN in ListImages/GetImage:

```sql
SELECT images.*, (favorites.image_id IS NOT NULL) AS favorited FROM images
LEFT JOIN favorites ON favorites.image_id = images.id
```

For `favorited=true` filter: `AND images.id IN (SELECT image_id FROM favorites)`
For `favorited=false` filter: `AND images.id NOT IN (SELECT image_id FROM favorites)`

### nsfw mapping (reconciliation note)

Schema CHECK: `nsfw IN ('sfw','nsfw','unknown')`.

- `sfw_only` -> `WHERE nsfw = 'sfw'` (excludes 'unknown')
- `nsfw_only` -> `WHERE nsfw = 'nsfw'`
- `all` -> no filter (includes 'sfw', 'nsfw', 'unknown')

### tags_source vs tags_user split

- `images.tags_source` = JSON array on the images row (source-supplied tags). Typed via `jsonCol<string[]>`.
- `image_user_tags` = junction table with `(image_id, tag)` PK. User-added tags.
- DTO exposes BOTH `tags_source: string[]` and `tags_user: string[]`.
- `AddTag` / `RemoveTag` operate on `image_user_tags` only.

### Pagination cursor fields

Images don't have `created_at`. The sort key is `ingested_at`. The existing `Cursor` type in
`src/lib/server/http/pagination.ts` uses `{ created_at, id }` -- we need to use `ingested_at` for
images. Two options:

1. Encode `ingested_at` into the cursor `created_at` field (reuse the existing type, same semantics).
2. Create an `ImageCursor` type.

Decision: reuse `Cursor` with `created_at` field storing `ingested_at` value. This is pragmatic --
the cursor is opaque to clients and never inspected except at decode time.

### ordering without search

`ORDER BY ingested_at DESC, id DESC` per reconciliation notes and api.md §Gallery.

### FTS rowid vs UUID join strategy

Since FTS uses `content_rowid='rowid'` and images uses a UUID text PK, the join must go through rowid:

```sql
AND images.rowid IN (SELECT rowid FROM images_fts WHERE images_fts MATCH ?)
```

When search is active, order becomes:

```sql
ORDER BY (SELECT bm25(images_fts) FROM images_fts WHERE images_fts.rowid = images.rowid) ASC, images.id ASC
```

### RestoreImage semantics

RestoreImage clears `deleted_at` only (not `blacklisted_at`). If `blacklisted_at IS NOT NULL`, throw
`AppError("validation.blacklisted", { status: 400 })`. Users must not be able to restore a blacklisted image.

### BlacklistImage clears favorites + user_tags

BlacklistImage must:

1. Set `blacklisted_at = Date.now()`
2. `DELETE FROM favorites WHERE image_id = ?`
3. `DELETE FROM image_user_tags WHERE image_id = ?`
   All in a single transaction.

### Device images junction

`device_images` table: `(device_id PK, image_id PK, on_disk_path, linked_at)`. For `ListDeviceImages`,
filter by `device_id` using an IN-subquery (same pattern as `device_id` filter in `ListImages`):
`AND images.id IN (SELECT image_id FROM device_images WHERE device_id = ?)`.

### Seed image IDs

For seed_images.ts, use literal UUIDv7-shaped strings so tests can assert specific rows. Using format:
`01900000-0001-7000-8000-000000000001` (fictional but well-formed UUIDs for testing).

## Invocation 2 observations

### Fixes applied

1. **`images.rowid` → `images.rowid` raw SQL**: Drizzle doesn't expose `rowid` as a typed column. The FTS IN-filter and bm25 ORDER BY both required bare SQL. Furthermore, with the LEFT JOIN to `favorites`, the bare `rowid` was ambiguous — SQLite rejects "ambiguous column name: rowid". The fix: `sql\`images.rowid IN ...\``and`sql\`... images_fts.rowid = images.rowid\`` both qualify with the table name.

2. **`_list_images_page1` private method removed**: The private helper method `_list_images_page1` was accessing `this.deps.db` via `(typeof this)["deps"]["db"]` as a parameter type, which TypeScript rejected because `this` isn't resolved inside a private method on a generic mixin class. Fix: all three call sites (cursor=null else-branches for next/prev/none) now inline the identical select block. Code duplication is acceptable here.

3. **`ListDeviceImages extends ListImages(Sup)` → `extends Sup`**: The `index.ts` composition chain already places `ListImages` before `ListDeviceImages`, so `this.listImages` is available on `this` via the mixin chain. Extending `ListImages(Sup)` again created a double-wrap. To call `this.listImages`, a `self` cast is used: `const self = this as unknown as { listImages(...): ... }`.

4. **18 test files updated** with `import { ImageService } from "$lib/server/service/images"` and `images: new ImageService({ db })` added to the services stub. Files span: 9 subscription service tests, 1 scheduler test, 8 route tests.

5. **`seed_images.ts` inserts devices first**: The `device_images` table has a FK on `device_id → devices(id)`. The seed fixture must insert DEVICE_A and DEVICE_B rows into `devices` before inserting `device_images` rows. Added `db.insert(devices).values([...]).run()` at the top of `seed_images()`.

### bm25 ordering direction confirmed

bm25() returns negative scores. ORDER BY bm25 ASC = best match first (least-negative). Confirmed: `asc(bm25_order)` in the search branch. Tests verified that "cat" search returns i18 (search_text "cat on a roof") and excludes i19 ("dog in the park").

### AppError assertion pattern (for future tests)

`toSatisfy((e) => e instanceof AppError && e.status === 404)` does NOT work in Bun test — the predicate returns false even when the error is a genuine `AppError`. The correct pattern used in the codebase is:

```ts
try {
  await svc.someOp(...)
  throw new Error("expected to throw")
} catch (e) {
  const err = AppError.is(e, AppError)
  expect(err).toBeDefined()
  expect(err?.status).toBe(404)
}
```

### ListImagesRequest offset/limit required fields

`z.infer<typeof ListImagesRequestSchema>` infers `offset: number` and `limit: number` as required (not optional) because `.default()` without `.optional()` doesn't make the TypeScript INPUT type optional. All service test calls must pass `offset: 0, limit: N` explicitly.

### ListDeviceImages composition issue

`ListDeviceImages` currently tries to extend `ListImages(Sup)` inside the mixin function body,
which is unusual and may cause diamond-inheritance issues. The `index.ts` for images domain
already includes both `ListImages` and `ListDeviceImages` in the composition chain. The safer
approach: `ListDeviceImages` just calls `this.listImages(...)` - it should NOT extend `ListImages(Sup)`
directly inside the mixin. Instead have the `index.ts` compose them in the right order so
`ListDeviceImages` can call `this.listImages` because ListImages is earlier in the chain.

Current `index.ts` composition: `... ListDeviceImages(GetImage(ListImages(Base) ...` which means
ListImages(Base) is there, then GetImage wraps it, then ListDeviceImages wraps that. So
`listImages` IS available on `this` when `listDeviceImages` is called. The issue is that
`ListDeviceImages` mixin body extends `ListImages(Sup)` creating a double-wrap. Fix: remove
the `extends ListImages(Sup)` and just `extends Sup` in the ListDeviceImages mixin. The method
is available via the chain in index.ts.

## Invocation 3 observations

### Route file pattern matched existing layout

All 7 route files follow the existing pattern: `import { runtime_ref }`, `try/catch` wrapping
every handler, `app_error_to_response(err)` in the catch, no business logic at the route layer.
The SvelteKit `event.params.*` fields are `string | undefined` in the type system — the pattern
used throughout the codebase is `event.params.id!` (non-null assertion) to satisfy TypeScript.

### parse_pagination + extra image filters

`parse_pagination(sp)` handles `limit/offset/next/prev`. Image-specific filters (`device_id`,
`source_slug`, `favorited`, `nsfw`, `include_deleted`, `include_blacklisted`, `search`) are parsed
manually. Boolean flags (`favorited`) use the `"true"/"false"` string comparison pattern seen in
devices/subscriptions routes. `NsfwFilterSchema.safeParse(nsfw_raw)` validates the enum value
before passing to the service.

### RestoreImage AppError message fixed

The service `RestoreImage.ts` was throwing `AppError({ message: "cannot restore blacklisted image: ..." })`
but IMPLEMENTATION.md specifies `AppError("validation.blacklisted", ...)`. Fixed to use
`message: "validation.blacklisted"` so the route's error code is machine-readable per the API contract
(and status_for_code in errors.ts maps it to 400 via the explicit `status` field).
