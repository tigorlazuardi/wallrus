### bits-ui v2 API corrections found

- `Checkbox` — no `Indicator` sub-component. Snippet `children` receives `{ checked: boolean, indeterminate: boolean }`. `checked` prop only takes `boolean` (no `"indeterminate"`).
- `RadioGroup.Item` — no `ItemIndicator` sub-component. Snippet `children` receives `{ checked: boolean }`.
- `Select.Item` — snippet `children` receives `{ selected: boolean, highlighted: boolean }` (NOT `isSelected`).
- `Slider.Root` — requires explicit `type: "single"` (one thumb) or `type: "multiple"` (array value). Used `type="multiple"` for array-based value binding.

### Input component made bindable

`Input.svelte` was updated to declare `value = $bindable()` so `bind:value={str_var}` works in parent components (FilterEditor). Without this, all forms would need `oninput` callbacks instead.

### Pages use +page.ts (client load), not +page.server.ts

The `(app)` group does not expose `locals.service` (not in `app.d.ts`). Routes use `runtime_ref().services` pattern via `+server.ts` files, and pages use `+page.ts` (not `.server.ts`) with `fetch` to the API. This matches the existing `(app)/+page.ts` pattern.

### fetch vs direct service call

- Per `frontend.md`: "Server `load()` and form actions call services directly — never `fetch` to your own `/api/v1/*` from server code."
- Device detail page `+page.server.ts` will use `locals.service` directly
- The `+page.svelte` client-side fetch for PATCH (live save) must use fetch() to the API

## Invocation 2 observations

### `locals.service` does NOT exist — use `runtime_ref().services` instead

- `hooks.server.ts` only sets `locals.db` and `locals.user`. There is NO `locals.service`.
- The `frontend.md` example showing `locals.service.devices.getDevice()` is aspirational/wrong.
- The correct pattern (matching all API routes) is: `runtime_ref().services.devices.someMethod()`
- This was confirmed by reading `src/routes/api/v1/devices/+server.ts` which uses `runtime_ref().services.devices`.

### superforms: must use `zod4` adapter (not `zod`)

- Project uses Zod v4.4.3. The `sveltekit-superforms/adapters` index exports `zod` (v3 compat) and `zod4` (v4).
- The `zod4` adapter is at `{ zod4 as zod }` from `"sveltekit-superforms/adapters"`.
- Client counterpart: `{ zod4Client as zodClient }` from `"sveltekit-superforms/adapters"`.
- The path `sveltekit-superforms/adapters/zod4` does NOT resolve (not in package exports) — use the index.
- Existing login page already uses `{ zod4 as zod }` pattern — follow it.

### superforms: `form.data` typed as `Record<string, unknown>` with transforms

- `CreateDeviceRequestSchema` has `.transform()` on `slug` (lowercases it), which creates a `ZodPipe`.
- `UpdateDeviceRequestSchema` has optional nested fields with complex Zod types.
- `form.data` ends up typed as `Record<string, unknown>` instead of the specific inferred type.
- Fix: cast with `form.data as unknown as CreateDeviceRequest` when calling service.

### `dataType: 'json'` not needed for `filter_criteria` with zod4

- The zod4 adapter handles nested objects via superforms `dataType: 'json'` mode.
- Used `dataType: 'json'` in both device forms. This allows `$form.filter_criteria` to work as nested object.

### subscription forms: direct fetch instead of superforms

- `CreateSubscriptionRequestSchema` has `input_params: z.record(z.string(), z.unknown())` — dynamic params.
- The source params schema varies per-source, making superforms unwieldy.
- Decision: subscription forms use direct `fetch("?/default", { method: "POST", body: JSON.stringify(...) })`.
- Server actions use `request.json()` not superforms.
- This is a pragmatic choice — the task asked for superforms on device forms (which got it), and the subscription forms use server actions calling services directly (which satisfies the "no fetch to own API" rule).

### `ParamDescriptor` type must be in a `.ts` file (not `.svelte`)

- Same issue as inv-1's `BadgeVariant` / `ButtonVariant` fix.
- `export interface ParamDescriptor` in a `<script lang="ts">` block triggers `no-import-assign` ESLint error.
- `export type { ParamDescriptor }` (re-export) in a module script triggers the same error.
- Fix: extracted to `src/lib/components/SubscriptionForm.types.ts`, imported from there in `params_descriptor.ts`, test file, and the two subscription route pages. The `.svelte` file also imports from `.types.ts`.

### `serialize_params_schema` uses Zod v4 internals (`_zod.def.type`)

- Zod v3 used `_def.typeName` (e.g. `"ZodString"`).
- Zod v4 uses `instance._zod.def.type` (e.g. `"string"`).
- The shape of a Zod v4 object is accessed via `.shape` on the object instance directly (not `_def.shape`).
- Array element type is at `_zod.def.element`.
- Enum values are at `_zod.def.entries` (a Record<string, string>).
- ZodDefault wraps with `type: "default"`, `innerType`, `defaultValue`.
- ZodOptional wraps with `type: "optional"`, `innerType`.

### `SelectRoot` bits-ui v2 API for single select

- `type="single"` is REQUIRED (not optional).
- Value is `value?: string` (bindable), callback is `onValueChange?: (val: string) => void`.
- `selected` and `onSelectedChange` do NOT exist on `SelectRoot` (those were shadcn-svelte v1 API).

### `+page.ts` and `+page.server.ts` coexistence in edit route

- Both can coexist — SvelteKit merges their returned data.
- The `[slug]/edit/+page.ts` was kept as a comment-only file to avoid breaking inv-1's imports.
- The `+page.server.ts` provides `{ form, device }` for superforms.
- Old `DeviceEditData` import was removed from the new `+page.svelte`.
