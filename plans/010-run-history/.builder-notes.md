# Builder notes — 010-run-history

## Invocation 1 observations

### Migration (0003)

- `bunx drizzle-kit generate --custom --name run_history_prune_trigger` auto-generated the file AND updated `_journal.json` with idx=3 and a timestamp. No manual journal edit needed (drizzle-kit handles it).
- The existing `idx_runs_sub_started` index in `0000_initial_schema.sql` covers only `(subscription_id, started_at)` without `id`. The new migration adds `idx_runs_sub_started_id ON run_history(subscription_id, started_at DESC, id DESC)` to support the trigger's subquery ORDER BY.
- SQLite trigger syntax: uses backtick-quoted identifiers for STRICT-table columns; `NEW.subscription_id` works without quoting.

### Bus (bus.ts)

- Uses `node:events` `EventEmitter`. Set `setMaxListeners(500)` to handle many concurrent SSE clients without the default 10-listener warning.
- `emit_update` / `subscribe` are module-level exports (not class methods) — this is a singleton bus pattern matching the SSE design in IMPLEMENTATION.md.

### update_run (update.ts)

- Uses `.returning()` on the Drizzle UPDATE — returns the full updated row atomically.
- Throws `AppError({ status: 404 })` when 0 rows returned (not found).
- Calls `emit_update(updated)` AFTER the DB write completes, guaranteeing SSE clients see a consistent row.

### Pipeline refactor

- Only one place needed changing: the `finalize_run` block (the single `db.update(run_history)...` call at the end of `run_subscription`). The initial INSERT stays raw — `update_run` is only for post-insert mutations.
- Import added at top of pipeline.ts: `import { update_run } from "../runs/update"`.
- No `withQueryName` wrapper needed around `update_run` — it wraps internally.

### Service composition quirk — ListSubscriptionRuns

- `ListSubscriptionRuns` delegates to `this.listRuns` via `(this as any).listRuns(...)`. This avoids double-wrapping ListRuns in the mixin chain while still having `listSubscriptionRuns` as a named method. The `eslint-disable` comment is on the cast line.
- Composition order in `runs/index.ts`: `ListSubscriptionRuns(GetActiveRuns(GetRun(ListRuns(Base))))` — ListSubscriptionRuns must be outermost so it can reach `listRuns` at runtime.

### Runtime stub updates in existing tests

- Adding `runs: RunService` to the `Runtime.services` type broke 26+ existing test files that built minimal Runtime stubs. All fixed by adding `runs: {} as never` to the stub objects.
- sed was used to batch-add the field to files where it appeared at end-of-line. Single-line inline stub objects (`{ devices, subscriptions, images }`) had to be edited individually.
- Prettier reformatted some of these files (the `runs: {} as never` addition triggered line-length violations in some inline objects, which prettier split to multi-line).

### Schema types

- `ListRunsRequest` (the inferred output type with defaults applied) has `offset: number` and `limit: number` as **required** fields. This mirrors the pattern in `ListDevicesRequest`. Tests must pass explicit `offset: 0, limit: 50`.
- `z.string().uuid()` rejects UUIDs whose 4th group doesn't start with `[89abAB]` (variant bits). Test UUIDs need to be valid RFC UUIDs — `018f9d5a-8c3e-7a2b-9d4e-5f6a7b8c9d0e` works.

### Cursor design for runs

- Runs list orders by `started_at DESC, id DESC`. The existing `encode_cursor`/`decode_cursor` in `src/lib/server/http/pagination.ts` uses `{ created_at, id }`. Rather than overload it, `ListRuns.ts` defines its own local `encode_run_cursor`/`decode_run_cursor` with `{ started_at, id }`.

## Invocation 2 observations

### Routes

- `[id]/+server.ts` uses `event.params.id ?? ""` to handle the `string | undefined` TS type from SvelteKit's `RequestEvent` params. The `?? ""` will still produce a 404 from `getRun` (not-found path), so behaviour is correct.
- `runs/+server.ts` and `subscriptions/[id]/runs/+server.ts` parse query params manually before calling `ListRunsRequestSchema.safeParse()` — numbers come in as strings from `URLSearchParams`, so explicit `Number()` conversion is needed.
- Subscription inserts in route tests must NOT include `updated_at` — that column does not exist on the `subscriptions` table schema (svelte-check caught this; fixed before gates ran).

### SSE route

- `stream/+server.ts` uses named export `GET` (not `RequestHandler` const) because SvelteKit accepts either and the named function avoids an unnecessary type annotation.
- `try { controller.close() } catch {}` — the spec says `close()` on an already-closed controller throws; we swallow since the client is already gone.

### SSE test pattern

- SSE test constructs a `Request` with `AbortController.signal`, calls `GET({ request })`, reads the `ReadableStream` body via `getReader()`.
- A 10ms `setTimeout` between call and `emit_update` gives the stream a micro-tick to register its `subscribe` listener before the first event fires.
- `read_chunk_with_timeout()` helper: races `reader.read()` against a 500ms fallback so the test doesn't hang if the event never arrives.
- Abort-closes test: after `abort.abort()`, we race `reader.read()` against a 500ms timeout and assert `done === true`.
- Keepalive ping test: marked `[-]` (skipped). Advancing `bun:test` fake timers inside a `ReadableStream.start()` callback causes races with the internal stream event loop. Behaviour is documented in the test file.

### Smoke

- Ran on port 5190 (not 5173) with `WALLRUS_AUTH_ENABLE=false WALLRUS_MODE=prod WALLRUS_LISTEN_ADDR=0.0.0.0:5190 WALLRUS_DATA_DIR=/tmp/wallrus-010-smoke`.
- All four endpoints responded correctly: 200 empty list, 200 active empty, 404 unknown id, 200 SSE with correct headers.
