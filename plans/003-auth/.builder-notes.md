# 003-auth builder notes

## Invocation 1 observations

### Env changes

- `parse_env` is synchronous but `Bun.password.hash` is async. The env Zod schema uses
  `.transform()` which is synchronous, so the hash must be computed outside the Zod transform.
- Solution: `parse_env` stays sync and returns the Zod-parsed object unchanged; we add a new
  async `parse_env_with_hash()` that calls `parse_env()` then computes the hash. The env Env
  type gains an optional `password_hash?: string` field via the Zod schema using `z.string()
.optional()` on `WALLRUS_PASSWORD_HASH`, and a derived `password_hash` field is added on the
  output type.
- Actually the IMPLEMENTATION.md says "discard the plaintext from the returned object so it
  doesn't leak via inspect". This means we need to either use `omit` or delete the key from
  the parsed result after hashing. The superRefine guard needs to check EITHER
  WALLRUS_PASSWORD OR WALLRUS_PASSWORD_HASH is set (not just WALLRUS_PASSWORD).
- The Env type (returned by parse_env) should NOT include `password_hash` because it's async.
  Instead, we define a separate `ParsedEnv` type that extends `Env` with `password_hash`.
  But this complicates the singleton. Better: use `parse_env` sync, add `password_hash` as a
  separate derived field that is computed once and stored.
- Cleanest approach:
  1. Add `WALLRUS_PASSWORD_HASH` to the Zod schema as optional string.
  2. Fix `superRefine` to accept either WALLRUS_PASSWORD or WALLRUS_PASSWORD_HASH.
  3. Strip `WALLRUS_PASSWORD` from the returned object after Zod parse (via `Omit` trick).
  4. Add separate `async init_password_hash(env: Env)` in env.ts that computes and stores the
     hash in the singleton. Call it from bootstrap.ts after `boot()`.
  5. Alternatively, add `password_hash?: string` to the Env type as an optional field that is
     populated async in bootstrap.
- **Decision made**: simplest approach consistent with the task spec:
  - Add `WALLRUS_PASSWORD_HASH` to Zod (optional).
  - Fix superRefine to accept EITHER password var.
  - The Env type is synchronous (as-is from Zod).
  - Add `password_hash?: string` to the Env type (not part of Zod, set imperatively).
  - Add an async `hash_password_in_env(e: Env): Promise<Env & { password_hash: string }>` helper
    that is called from the env singleton or bootstrap.
  - For the singleton, the cleanest approach: change `_env` to hold a richer type that includes
    `password_hash`, and add `init_env()` async function that bootstrap calls once.
- WAIT: The prompt says "at parse time, if WALLRUS_PASSWORD_HASH is set use it directly; else
  if WALLRUS_PASSWORD is set hash it via await Bun.password.hash(...)". This implies parse_env
  becomes async or we change the bootstrap flow. The simplest: keep parse_env sync (validates),
  add a separate `async boot_env()` that calls parse_env then computes the hash and caches it.
- **Final approach**:
  - `parse_env` stays sync (for tests that don't need password_hash).
  - Add `WALLRUS_PASSWORD_HASH: z.string().optional()` to RawEnv.
  - Fix superRefine: password requirement = WALLRUS_PASSWORD OR WALLRUS_PASSWORD_HASH.
  - The Env type includes both optional vars.
  - Add a separate export: `async function init_password_hash(): Promise<void>` that:
    1. Calls `parse_env()` to get the raw env (or reads the singleton).
    2. Computes the hash.
    3. Stores it in `_password_hash` (separate module-level var).
  - Add `function password_hash(): string | undefined` getter.
  - Delete `WALLRUS_PASSWORD` from the parsed result so it doesn't leak -- but Zod types this
    as string | undefined, and TypeScript won't let us delete it from the typed object.
  - Practical approach: don't delete the key from the object (TypeScript won't like it), but
    document that password_hash() is the correct access path. Tests will verify plaintext
    isn't on the final parsed env if we choose Omit.
- **Simpler final approach** that satisfies TASKS.md exactly:
  - Add `WALLRUS_PASSWORD_HASH` to Zod.
  - The `Env` type returned by `parse_env` will have `WALLRUS_PASSWORD` and `WALLRUS_PASSWORD_HASH` as optional.
  - Add a derived field: after Zod parse, replace `WALLRUS_PASSWORD` with `undefined` on the
    returned object (delete key), and add a `password_hash?: string` field.
  - But this can't be async in sync `parse_env`! So the password_hash field will only exist
    if WALLRUS_PASSWORD_HASH is provided.
  - For the async hash case, we need `init_env` to be called at bootstrap.
  - This is the cleanest: parse_env is sync and just strips plaintext, boot_env is async and computes hash.

- Returns `Promise<boolean>`.
- For argon2id: `await Bun.password.verify(plaintext, hash)` — Bun auto-detects algorithm from hash format.
- `Bun.password.hash(plaintext, { algorithm: "argon2id" })` to hash.

### Rate limit: prune-on-check

- Store `Map<string, number[]>` where numbers are `Date.now()` timestamps.

## Invocation 2 observations

### env() singleton globalThis fix

The `env()` singleton originally used a module-level `let _env`. When `init_password_hash()` in
`bootstrap.ts` (executed in the cli.ts process context) mutated `_env.password_hash`, the SvelteKit
build artifact (`build/handler.js`) had its own compiled copy of `env.ts` with a separate `_env =
null`. On first request, the handler's `env()` called `parse_env()` again — but since `WALLRUS_PASSWORD`
was already stripped from the live process.env (Bun.env), `password_hash` remained undefined.

Fix: changed `let _env` to `globalThis.__wallrus_env__` (same pattern as `runtime.ts`). This ensures
both the cli.ts context and the SvelteKit handler share the same parsed+hashed env object.

### sveltekit-superforms Zod v4 adapter

This project uses Zod v4. The standard `zod` + `zodClient` imports from `sveltekit-superforms/adapters`
expect Zod v3 types. Use `zod4` and `zod4Client` instead:

```ts
import { zod4 as zod, zod4Client as zodClient } from "sveltekit-superforms/adapters"
```

### superforms message helper

Setting `form.message = "..."` directly requires the Message generic to be typed. Use the
`message(form, "text", { status: 4xx })` helper instead — it returns the correct `ActionResult` and
types cleanly.

### Svelte 5 `data.form` warning with superForm

`superForm(data.form, ...)` triggers a Svelte 5 lint warning "This reference only captures the initial
value of `data`". This is a false positive — `superForm` is designed to take the server snapshot as
initial state and manages its own internal reactive state thereafter. The warning is accepted.

### OTLP proxy test approach

No existing tests for the OTLP proxy route. Rather than standing up the full SvelteKit+Bun stack,
the tests exercise `otel_frontend_posture()` (the env-derived posture logic) and simulate the
handler's auth gate check (`auth_required && !locals.user`) as a pure function. This gives
meaningful coverage of the gate logic without requiring a live HTTP server.

### authenticate() return for Basic auth

When an `Authorization` header is present but fails verification (wrong password or unknown user),
`authenticate()` returns `null` immediately — it does NOT fall through to the cookie check. This
prevents an attacker from providing a bad Basic header while still getting through on a valid cookie,
which would be a confused-deputy issue.
