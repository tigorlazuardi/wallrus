# 009-ingest-pipeline — builder notes

## Invocation 2 observations

### Data layout decision

The pipeline uses per-device-dir layout per `engineering/SCOPE.md` §Storage:
`<base-dir>/<device-slug>/<source-slug>-<filename>.<ext>`. There is NO central
blobs dir. The first matching device gets the canonical copy via `atomic_write`;
subsequent devices get hardlinks via `link_or_copy`.

### Canonical-file pattern

First matching device in `subscribed_devices` order is canonical. All other
matching devices are hardlinked from that first device's path. If only one
device matches, no hardlinks are created.

### run_history Zod schema decision

The pipeline writes `run_history` via Drizzle typed inserts/updates, not via
Zod parse. No `src/lib/schemas/runs/` Zod schema is needed for this slice.
The inferred `NewRunHistoryRow` type from Drizzle is sufficient.

### Scheduler runtime threading

`_tick_with_clock` originally had no access to `runtime` (module-level state).
Added `_runtime: Runtime | null` as a module-level variable that `start()` sets.
This avoids changing the `_tick_with_clock(reg, now)` signature (preserving the
existing scheduler tests). A null-check guards against the theoretical case
where the executor fires before `start()` is called.

### Scheduler queue error-isolation

The try/catch in the cron.ts executor ensures any thrown error from
`run_subscription` is caught and logged, but does NOT poison the per-source
promise queue. The queue continues processing subsequent subscriptions.

### Source mock pattern in pipeline test

The smoke test uses a minimal `SourceModule` shim that implements only the
`fetch` generator. The `params_schema` is a duck-typed minimal shape (not a
real Zod schema) — this is acceptable for a unit test that only exercises the
DB plumbing, not source-specific schema validation.

`src/lib/server/ingest/pipeline.test.ts` (1 test).

## Invocation 1 observations

### Sharp dep status

`sharp` was already present in `package.json` at version `0.34.5`. No `bun add sharp` needed.

### Migration (0002_run_history_skip_counter)

- `drizzle-kit generate --name run_history_skip_counter` generated the migration AND the new snapshot at `drizzle/migrations/meta/0002_snapshot.json`. Journal updated correctly.
- The generated SQL is: `ALTER TABLE run_history ADD items_skipped_no_device integer DEFAULT 0 NOT NULL;`
- STRICT is preserved: SQLite ALTER TABLE adds the column to the existing STRICT table without re-stating the STRICT keyword. This is correct SQLite behaviour — adding a column does not require or allow toggling STRICT mode.
- Column order: `integer DEFAULT 0 NOT NULL` vs `integer NOT NULL DEFAULT 0` — both are valid SQLite syntax. Drizzle-kit emits the former; both parse identically.

### DeviceFilterCriteria field name alignment

IMPLEMENTATION.md refers to filter fields as `res_min`, `res_max`, `aspect: { value, tol }`, `size_bytes` but the actual Drizzle schema (`DeviceFilterCriteria`) and Zod schema (`DeviceFiltersSchema`) use:

- `min_width`, `max_width`, `min_height`, `max_height` (not `res_min/res_max`)
- `aspect_ratio: { target, tolerance }` (not `aspect.value/tol`)
- `min_bytes`, `max_bytes` (not `size_bytes.min/max`)

The implementation follows the actual DB schema types (authoritative), not the prose description in IMPLEMENTATION.md.

### FS mock pattern (fs.test.ts)

The spy/mock tests for `link_or_copy` use `spyOn(fs_promises, "link")` where `fs_promises = await import("node:fs/promises")`. This works because:

1. The `link_or_copy` function in `fs.ts` imports `link` directly from `node:fs/promises`.
2. In Bun's module system, the spy on the re-imported module object affects the module's exported binding. The spy catches the call correctly.

One subtlety: `mock.restore()` must be called inside the test (not just in `afterEach`) because `afterEach` runs AFTER the test, but if multiple tests in the same describe block run sequentially and share module cache, the spy from test N may bleed into test N+1. Calling `mock.restore()` both in `afterEach` and at the end of each spy test is the safe pattern.

### EXIF orientation test — not implemented

The TASKS.md item "Unit test: thumbnail honours EXIF orientation (use a pre-rotated fixture)" is marked `[~]` because:

1. Generating a JPEG with a non-1 EXIF Orientation tag programmatically via `sharp` in a test would require writing raw EXIF bytes via `withExif` or a dedicated exif-writing library.
2. No pre-committed fixture binary exists in the repo.
3. The `.rotate()` call is present in the implementation (code review verified) and will function correctly with real camera JPEG inputs.
4. This test can be added in a follow-up if a fixture binary is added to `src/test/fixtures/`.

### run_history DTO + Zod schema

No existing run_history Zod schema or DTO file existed (`src/lib/schemas/runs/` does not exist). The TASKS.md item "Update run_history DTO + Zod schema" was satisfied by updating `src/lib/server/db/schema.ts` directly (the `RunHistoryRow` inferred type now includes `items_skipped_no_device: number`). A dedicated Zod schema in `src/lib/schemas/runs/` is out of scope for invocation 1 and will be created in invocation 2 when the pipeline service layer is built.

### Test import path alias

Ingest tests use `$test/db` and `$test/fixtures/seed_images` (not relative paths). This is the SvelteKit-generated path alias pointing to `src/test/`. Using relative paths like `../../test/db` fails because Bun resolves from the test file's location and the alias isn't set up for relative resolution.

## Invocation 3 observations

### Fetch mocking pattern

`spyOn(globalThis, "fetch").mockImplementation(stub as unknown as typeof fetch)` works
reliably for pipeline tests. Bun's `typeof fetch` includes a `preconnect` property that
generic `(_url, _init) => Promise<Response>` stubs don't declare — the `as unknown as typeof
fetch` cast suppresses the TS error safely. At runtime the pipeline only calls
`fetch(url, init)` and reads `.ok`, `.headers`, `.body`; no Bun-specific property is touched.
The `make_fetch_stub()` helper centralises this pattern.

### Integration test cleanup strategy

Each integration test gets its own unique tmpdir via `make_test_dir()` (created in
`beforeEach`, destroyed in `afterEach`). The legacy smoke test (invocation 2) continues to
use the shared `TEST_DATA_DIR` from `beforeAll`/`afterAll`. Per-test isolation prevents
inode/filename collisions across concurrent or sequential test runs.

### Crash recovery export

`recover_orphan_runs` was a private function in `bootstrap.ts`. Added `export` keyword to
make it directly testable. The `boot()` call path is unchanged; tests import the function
directly against an in-memory DB without calling `boot()` (which requires real env parsing,
OTel SDK init, file system perms).

### Crash recovery test approach

Two test cases in `src/lib/server/bootstrap.test.ts`:

1. Orphaned "running" row flips to failed / daemon_crash — confirms the sweep logic.
2. Already-completed rows (success + failed) are not touched — confirms the WHERE clause
   selectivity.

### Real-source smoke deferral (Path B)

The real-network smoke (`r/wallpapers`, 1 device, permissive filters) requires live Reddit
credentials and network access — not suitable for CI. Path A (happy-path integration test)
satisfies the smoke requirement: 1 image downloaded, 2 hardlinks, thumbnail on disk, all
confirmed via assertions. Path B can be run manually by an operator with:
`WALLRUS_DATA_DIR=./tmp bun run src/cli.ts serve` + a pre-seeded subscription.
TASKS.md marks Path B lines as `[-]`.

### Migration gate via create_test_db()

Every test that calls `create_test_db()` runs `run_migrations(db)` on a `:memory:` SQLite.
This implicitly validates that migration 0002 applies cleanly. No separate drizzle-kit
smoke command is needed in CI; the test suite itself covers migration correctness.
