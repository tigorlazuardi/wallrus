# Builder notes ÔÇö 008-source-booru

## Implementation observations

### Credential schema: flat refine vs discriminated union

Chose the flat `z.object({ login?, user_id?, api_key })` with `.refine()` over
`z.discriminatedUnion("variant")`. Rationale: `variant` is already in
`BooruInputSchema` (the input params). Having a second `variant` discriminator in
the credential schema would require keeping them in sync at the call site and would
make anonymous callers specify a variant they don't need. The flat refine is
simpler: runtime picks the correct auth fields based on `input.variant`, and the
schema just validates that at least one of `login`/`user_id` is present alongside
`api_key`.

### Gelbooru wrapper: `post` (modern) vs `posts` (legacy)

Live Gelbooru returns `{ "@attributes": {...}, "post": [...] }`. Some older API
endpoints returned `{ "posts": [...] }`. The module handles both:

```ts
const gel = raw as GelbooruResponse
posts = gel.post ?? gel.posts ?? []
```

The gelbooru fixture uses the modern `post` shape. The legacy `posts` shape is
covered by an inline test (`gelbooru posts (legacy) wrapper also works`) that
constructs an inline `{ posts: [...] }` response without a dedicated fixture file.

### NSFW mapping

Per reconciliation notes (overriding the IMPLEMENTATION.md which erroneously
mentions `'safe'`):

- `s` (safe) Ôćĺ `sfw`
- `g` (general, newer danbooru) Ôćĺ `sfw`
- `q` (questionable) Ôćĺ `nsfw`
- `e` (explicit) Ôćĺ `nsfw`

Booru ratings are always present in the response, so `'unknown'` is never used.

### Tag normalisation

Tags are lowercased and deduped via a `Set<string>`. Empty string tokens (from
splitting whitespace-only strings) are filtered out. The `filter(Boolean)` call
on the concat removes empty tokens before the dedup set walk.

### `format` field and `'gif'` in extract_format

The `Format` type in `_types.ts` only allows `'jpg' | 'png' | 'webp' | 'avif'` ÔÇö
no `'gif'`. `extract_format` returns `"gif"` as a sentinel to trigger the skip
branch; the `'gif'` string is never assigned to the yielded `SourceItem.format`
field. Gif posts are silently skipped before the item is constructed.

### Aggregator wiring

Followed the pattern from slice 007: added `register_booru()` function in
`booru.ts` that calls `register(booru_module)`, and called it from
`register_sources()` in `_registry.ts`. No separate `_aggregator.ts` file ÔÇö
`register_sources()` is the aggregator (per reconciliation note from slice 007).

### `search_text` field

Booru items have no natural title. `search_text` is populated with the first 20
tags joined by space (consistent with the IMPLEMENTATION.md suggestion to use
"first 80 chars of joined tags"). This gives FTS5 coverage for the most
content-representative tags without indexing the full (potentially very long) tag
list.

### Danbooru `image_width`/`image_height` vs Gelbooru `width`/`height`

Field names differ between variants:

- Danbooru: `image_width`, `image_height`
- Gelbooru: `width`, `height`

Both are mapped to `item.width`/`item.height` at yield time. Nulls are omitted
(conditional assignment `if (post.image_width != null) item.width = ...`) so the
runtime can probe from bytes when absent.
