## Phase 5 (this invocation)

### No `images/[id]` route exists — wired into `ImageModal.svelte` instead

IMPLEMENTATION.md referenced `src/routes/(app)/images/[id]/+page.svelte` which does not exist.
The image detail UI is a modal (`ImageModal.svelte`). The wallpaper UI was wired into that
component instead. TASKS.md updated to reflect the actual file path.

### Detail-UI test approach — module interface only

Component rendering in `bun:test` is not possible (no jsdom, no Svelte compiler).
The wallpaper test (`wallpaper.test.ts`) verifies:
- `WallpaperPlugin` interface is callable with all 3 targets (home/lock/both)
- `registerPlugin("Wallpaper")` is called when the module is imported
- Success and failure paths on the return type

The `bun run check` gate exercises full Svelte compilation of `ImageModal.svelte`
including the wallpaper UI, providing type-safety coverage for the template.

### `WALLRUS_MOBILE_RELEASE_MANDATORY` — use `.default("false")` not `.optional()`

Initial impl used `.optional().transform(v => v === "true")` which produced `boolean | undefined`
in the Zod output type, breaking `otlp.test.ts`'s `make_env`. Changed to
`.default("false").transform(v => v === "true")` so the type is always `boolean`.
Also added `WALLRUS_MOBILE_RELEASE_MANDATORY: false` to `make_env` in `otlp.test.ts`.

## Phase 6 (this invocation)

### `boot()` return type changed — BootResult discriminated union

`boot()` now returns `BootResult` (discriminated union) instead of `"/setup" | "/"`.
Any caller (app root, setup page) that was doing `if (route === "/setup")` still works
since `result.route` is the discriminant. The setup page's `+page.ts` does not call
`boot()` directly (it uses `isNativePlatform()` for the 404 gate), so no breaking change.

### `APP_VERSION` constant — `src/lib/client/mobile/version.ts`

Standalone file so it can be mocked in `boot.test.ts` without affecting the module graph.
Pattern: `mock.module("./version", () => ({ APP_VERSION: "1.0.0" }))`.

### Release endpoint — public, no auth

`/api/v1/mobile/release/latest` added to `EXACT_ALLOWLIST` in `hooks.server.ts`
alongside `/api/v1/auth/status`. Route reads env directly (thin config read,
not business logic — no service file needed per the slice spec).

## Phase 3 (this invocation)

### `bun add @capacitor/core @capacitor/preferences`

`lefthook install` in the `prepare` postinstall script fails on NixOS because
`core.hooksPath` is set globally to the nix store. This is a pre-existing NixOS
issue — the packages ARE installed correctly (verified via `ls node_modules/@capacitor/`).
The error is from `lefthook install`, not from the package install itself.

### `use-auth-mutation.svelte.ts` — no `$state` runes

Existing mutation hooks in this project (e.g. `use-device-mutation.svelte.ts`) do NOT
use `$state` runes — they return plain action functions with no internal reactive state.
Callers manage loading/error state themselves. Followed the same pattern here.
Using `$state` in a `.svelte.ts` file fails in `bun:test` because the rune compiler
doesn't run for `.ts` files in the test runner context.

## Phase 4 (this invocation)

### Setup-screen component test — SKIPPED (documented)

`@testing-library/svelte` is installed in devDependencies, but the project has
NO vitest config (`vite.config.ts` has no `test:` block, no `vitest.config.ts` exists).
`bun:test` cannot run Svelte component tests — it does not understand `.svelte` files
or rune compilation.

The setup screen logic is almost entirely in the `<script>` block (test_connection,
submit_login, save_and_navigate). The critical paths are covered by:

- `use-auth-mutation.test.ts` — covers the login action that the setup page calls
- `boot.test.ts` — covers the Preferences read side
- The route's `+page.ts` 404 gate is trivial (single if-statement)

A full component test would require adding vitest + `@testing-library/svelte` runner
config (a separate slice concern). The TASKS.md line is marked `[-]` (skipped with
reason) rather than `[x]`.

### `+page.ts` — 404 gate

Returns `error(404, "Not found")` on web (non-native). Returns `{}` on native.
`isNativePlatform()` is the same mock seam used by boot and fetcher tests.

### `+page.svelte` — two-phase flow

Phase "url": URL input + "Test connection" button. Calls `/healthz` then `/api/v1/auth/status`.
Phase "auth": reveals username + password fields (auth_enabled=true path).
Phase "done": navigation target after save.
No auth_enabled=false path shows credentials — goes directly to `save_and_navigate`.
