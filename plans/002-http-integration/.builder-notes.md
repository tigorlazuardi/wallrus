# Builder notes — 002-http-integration

## Invocation 1 observations

### getLogger() signature

The actual `@tigorhutasuhut/telemetry-js/bun` export has `getLogger(): Logger`
(zero args). The rule docs mention `getLogger({ module: "..." })` but that
module context is passed as an **attribute on log calls** instead. Pattern used
throughout this slice: `log.info("msg", { module: "scheduler", ... })`.

### WALLRUS_MODE env var

Added `WALLRUS_MODE: z.enum(["prod","dev"]).default("prod")` to `src/lib/server/env.ts`.
The gate in `cli.ts serve` (invocation 2) checks:
`env.WALLRUS_MODE === "prod" || process.env.NODE_ENV === "production"`.

### parse_listen_addr edge case

`lastIndexOf(":")` is used, which handles IPv6-like strings gracefully (last
colon separates host from port). Empty host before colon defaults to `0.0.0.0`.

### queue.ts — pending_chains removed

The pre-existing stub exported `pending_chains()`. Invocation 1 replaced that
with `wait_idle()` as the slice requires. If any caller used `pending_chains`,
they need to be updated (grep found none in the codebase).

### cron.ts — \_tick_with_clock export

Exported as `_tick_with_clock(reg, now)` for direct test use. The `start()`
function uses the module-level `_registry` which is NOT accessible to external
tests (so tests build their own `Map<string, CronEntry>` and pass it directly).
The `CronEntry` type is also exported.

### runtime.ts — \_reset_runtime_for_tests

Exported `_reset_runtime_for_tests()` to allow afterEach cleanup in tests.
Production code must NEVER call this.

### boolCol and Drizzle .all() return type

`subscriptions.enabled` is a `boolCol`, but `.select({ enabled: subscriptions.enabled })`
returns the JS boolean. `eq(subscriptions.enabled, true)` works correctly because
Drizzle's customType `toDriver` is invoked in the query builder.

## Invocation 2 observations

### runtime.ts must use globalThis, not module-level singleton

`cli.ts` runs from source (`bun run src/cli.ts`), but the SvelteKit handler
runs from the build output (`build/server/chunks/hooks.server-*.js`). These
are two distinct module instances. A plain `let _runtime = null` singleton
set in the source `runtime.ts` is invisible to the built copy. Fixed by
storing on `globalThis.__wallrus_runtime__` so both module instances read
from the same global. Any future module with singleton state that must survive
the cli → build boundary needs the same pattern.

### svelte-adapter-bun exports `getHandler()`, not a default function

The build output `build/handler.js` exports `getHandler(): { fetch, websocket }`
(NOT a default export of a bare `fetch` function). Spread into `Bun.serve()`:
`Bun.serve({ port, hostname, ...getHandler() })`. Trying to pass the function
directly as `fetch` fails with `ERR_INVALID_ARG_TYPE`.

### Dynamic import path via URL prevents stale @ts-expect-error

Using `new URL("../build/handler.js", import.meta.url).href` as the import
path prevents TypeScript from statically resolving the build artifact. This
avoids the "unused @ts-expect-error directive" error that occurs when the
artifact exists post-build. The `as any` cast on the spread satisfies
TypeScript without triggering the TS resolve path.

### tsconfig.json must explicitly override include to exclude build/

After `bun run build`, the `build/` directory contains JS files. The default
`.svelte-kit/tsconfig.json` `include` list doesn't cover `../build/**`, but
`svelte-check` scans it anyway when `allowJs: true` + `checkJs: true` are
set (likely via `rootDirs`). Fixed by overriding `include` in the top-level
`tsconfig.json` to mirror the `.svelte-kit/tsconfig.json` list but rooted at
the project root. This is necessary for `bun run check` to remain clean after
`bun run build`.

### Test file location for SvelteKit routes

Test files (`*.test.ts`) cannot live inside `src/routes/` if their name starts
with `+` (e.g., `+server.test.ts`). SvelteKit considers all `+`-prefixed
files as route files. Route handler tests must live outside `src/routes/`.
Used `src/lib/server/healthz.test.ts` with a relative import to the handler.

### db.close() on Drizzle client requires .$client.close()

`Runtime.db` is a Drizzle `BunSQLiteDatabase`. The underlying `bun:sqlite`
Database is accessible at `db.$client`. Drizzle doesn't proxy `.close()`.
Shutdown sequence: `await runtime.sdk.shutdown()` then `runtime.db.$client.close()`.

### Locals.db type in app.d.ts

The `db` field in `Locals` should be typed as `import("./lib/server/db/client").DbClient`
(relative path from `src/app.d.ts`). The full generic Drizzle type with the
project schema is `DbClient`. Using `import("bun:sqlite").Database` (as the
spec suggested) would be a type mismatch since `runtime.db` is a Drizzle client.
