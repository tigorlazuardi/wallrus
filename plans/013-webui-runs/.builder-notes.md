# .builder-notes.md — 013-webui-runs

## SSE event payload shape

- Endpoint `GET /api/v1/runs/stream` emits only **one** event type: `event: run.update`
- Payload is a raw `RunHistoryRow` (Drizzle row), NOT the Run DTO.
  - The SSE `bus.ts` emits `RunHistoryRow` not the `Run` DTO.
  - The `stream/+server.ts` sends: `event: run.update\ndata: ${JSON.stringify(run)}\n\n`
  - The client receives a `RunHistoryRow`, which has the same shape as `Run` but may differ.
  - Decision: client parses SSE events as `Run` DTO (same fields via `to_run_dto` mapping in base.ts).
  - `RunHistoryRow` has `status` as `"running" | "success" | "failed"` — NO `daemon_crash` status.
  - `daemon_crash` is a `stop_reason` value, not a `status`. The IMPLEMENTATION.md's badge note about
    `daemon_crash` as a status is inaccurate. The slice spec says `status: 'running'|'success'|'failed'|'daemon_crash'`
    but the actual DB/schema only has 3 statuses. Decision: implement badge with the 3 real statuses;
    show stop_reason for daemon_crash context in RunDetail.

## Reconciliation: `locals.service` does NOT exist

## Run DTO does NOT include subscription_slug/source_slug

- `Run` schema only has `subscription_id`. Pages that show subscription details need to either:
  1. Do a separate service call to get subscription info, or
  2. Have the page.server.ts fetch a subscriptions map and pass it alongside runs
- Decision: page.server.ts fetches both runs (ListRuns) and subscriptions (ListSubscriptions with large limit)
  to build a `Map<subscription_id, { name, source_slug }>` for the RunRow component.
- For `/subscriptions/[id]/runs`, the subscription is already known from params.

## Status enum vs stop_reason

- `Run.status`: `"running" | "success" | "failed"` (3 values, matches DB schema)
- `Run.stop_reason`: `"max_items_inspected" | "source_exhausted" | "error" | "daemon_crash" | null`
- Badge colors: running=blue+pulse, success=green, failed=red
- daemon_crash is shown as additional context in RunDetail, not as a badge variant

## bun:test pattern for SSE stream tests

- Use `Object.defineProperty(globalThis, 'EventSource', ...)` for mocking
- Fake timers: bun:test exposes `mock.setSystemTime()` but NOT `useFakeTimers()` for setInterval.
  Decision: mock setInterval/clearInterval/setTimeout/clearTimeout via `Object.defineProperty(globalThis, ...)`
  or extract timer logic as injectable functions for testability.

## `runtime_ref()` import path

- Confirmed from existing routes: `import { runtime_ref } from "$lib/server/runtime"`

## Injectable Deps pattern in runs-stream.ts

- Used `type Deps` with simple `number` timer IDs instead of `ReturnType<typeof setTimeout>` to avoid
  complex Bun-specific types that can't be easily mocked in tests.
- DEFAULT_DEPS wraps globalThis.setInterval/setTimeout with `as unknown as number` casts.
- Test uses plain functions (not `mock()`) for timers to satisfy the `(cb: () => void, delay: number) => number` signature.
- Tests create a `FakeEventSource` class with an `instances` static array and direct trigger helpers.

## AppError.is() return pattern

- `AppError.is(err, AppError)` returns `AppError | null` (not boolean).
- Pattern: `const app_err = AppError.is(err, AppError); if (app_err && app_err.status === 404) { ... }`

## listSubscriptions requires include_deleted

- `ListSubscriptionsRequestSchema` has `include_deleted: z.boolean().optional().default(false)`.
- Inferred type requires the field. Pass `include_deleted: true` in runs page server load to also
  show subscription names for runs that belong to soft-deleted subscriptions.

## svelte/prefer-writable-derived ESLint rule

- `let x = $state(...) + $effect(() => { x = ... })` triggers this rule.
- Fix for detail page: use `let sse_run = $state<Run | null>(null)` + `const run = $derived(sse_run ?? data.run)`.
  SSE sets `sse_run` directly; `$derived` combines both sources without the rule trigger.
