## Environmental floor: `bun test` has 33 pre-existing sharp failures

On this machine `bun test` reports **33 failures, all from `sharp`**
(`ERR_DLOPEN_FAILED: libstdc++.so.6: cannot open shared object file`).
The native binary can't load in this environment. These are:

- **Pre-existing** — present before slice 015 started.
- **Unrelated to 015** — every failing file is an image/thumbnail/ingest
  test that imports sharp. Slice 015 (client hooks + universal load + SPA
  forms) introduces zero sharp usage.
- **Not lefthook-gated here** — `git config core.hooksPath` points at a
  Nix store dir whose only hook is `commit-msg`; there is no `pre-commit`
  and `lefthook` is not on PATH, so commits are not blocked locally.

Reviewer verification each iteration therefore uses the green code gates
plus the slice's own new tests, and treats the sharp 33 as a known floor:

- `bun run check` → 0 errors (4 pre-existing Svelte `state_referenced_locally` warnings)
- `bunx eslint .` → 0 errors (1 pre-existing `no-explicit-any` warning in `service/base.ts`)
- `bunx prettier --check .` → clean
- new slice tests → must pass with 0 fail
- `bun test` total → expect `554+ pass, 33 fail` where the 33 are ALL sharp.
  If a NON-sharp failure appears, that IS a regression — fix before commit.

If sharp ever gets fixed in this env (libstdc++ available), the floor
drops to 0 and the full suite must be green.
