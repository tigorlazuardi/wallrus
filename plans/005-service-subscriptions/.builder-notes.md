# .builder-notes.md — 005-service-subscriptions

## Invocation 1 observations

### Registry naming reconciliation

The slice spec (`IMPLEMENTATION.md` "Resume here") refers to `register(entry)`, `lookup(slug)`, `list()` but the file `src/lib/server/sources/_registry.ts` already exists with `sources: Record<string, SourceModule>`, `get_source(slug)`, `list_sources()`. Per reviewer instructions: **use existing API names**. No rename. We add `register_sources()` as an aggregator stub (no-op today; 007/008 fill it in). We also add `register(entry)` as a helper that inserts into `sources` map (needed by tests to seed mock sources). The TASKS.md entry says "register(entry), lookup(slug), list()" — we interpret "lookup" = `get_source` and "list" = `list_sources`, and add `register(entry)` + `register_sources()`.

### Scheduler reload: sync → async

`cron.ts` currently exports `reload(runtime: Runtime): void`. TASKS.md specifies `reload(runtime): Promise<void>`. We made it `async` (just adds `async` keyword — `load_registry` is sync, no actual await). Low-risk. Services `await scheduler.reload(runtime_ref())` after commit.

### How services reach `runtime` for reload

Each mutation mixin (`Create/Update/Delete/Toggle`) imports `runtime_ref` from `$lib/server/runtime` and calls it after the DB commit. In tests, `_reset_runtime_for_tests()` + `set_runtime({ db, ... })` wires a minimal Runtime. The scheduler `reload` is safe when `_interval === null` (just rebuilds the map).

### Mock source pattern in tests

Tests that exercise the happy path of `CreateSubscription` need a registered source. We use:

```ts
import { sources } from "$lib/server/sources/_registry"
beforeEach(() => {
  sources["mock"] = {
    slug: "mock",
    display_name: "Mock",
    params_schema: z.object({}).passthrough(),
    async *fetch() {},
  }
})
afterEach(() => {
  delete sources["mock"]
})
```

This mutates the module-level `sources` map and is restored after each test.

### `subscriptions` table columns confirmed

From `schema.ts`: `id, source_slug, name, input_params, cron, enabled, max_items_inspected, created_at, deleted_at`. Note: `name` column exists (required, not in the IMPLEMENTATION.md Decisions but present in schema). Must include in `CreateSubscriptionRequestSchema`.

### `device_subscriptions` junction columns

`device_id, subscription_id, created_at`. PK is `(device_id, subscription_id)`.

### Cron validation

Done inside `CreateSubscription` service with `new Cron(expression, { paused: true })`. In the wire schema (`CreateSubscriptionRequestSchema`) we also add a `.refine()` using croner — but since croner is a server-only package, we must NOT import it from `src/lib/schemas/`. Solution: cron validation in wire schema is skipped (no croner in schemas); the service does the validation. The schema just validates it's a non-empty string. Unknown source validation via `get_source(slug)` refine IS fine (sync) since `_registry.ts` is not server-only.

Wait — re-reading the instructions: "CreateSubscriptionRequestSchema should validate cron via croner inside a .refine()". But croner must not be imported in `$lib/schemas/`. Reviewer said: "catch error, mark invalid". The resolver says we can only put pure `zod` in schemas. So cron validation stays in the service only. The schema just validates `cron: z.string().min(1)`.

For `source_slug` refine: `get_source` is importable from `$lib/server/sources/_registry` which is server-only. Schemas must stay universal. So source_slug refine CANNOT be in the universal schema — it must be in the service. The schema just validates `source_slug: z.string().min(1)`.

Summary: The wire schema is simple (no refines that call server-only code). Validation of cron and source_slug happens inside the service operations.

### input_params in wire schema

`z.record(z.string(), z.unknown())` — option (a) per reviewer guidance. Service validates per-source params after looking up the source.

## Invocation 2 observations

### Route test directory naming

Test files for subscription routes live under `src/test/routes/api/v1/subscriptions/` with literal directory names `id/`, `id/toggle/`, `id/devices/`, `id/devices/device_id/` — NOT bracket names like `[id]/`. Bracket-named test dirs confuse the bun test scanner and would collide with SvelteKit's route tree convention. The handler imports use relative paths to `routes/api/v1/subscriptions/[id]/+server.ts` etc. (actual SvelteKit bracket paths) — so the import paths use brackets but the test file's parent dir does not.

### ListSubscriptionDevices needs pagination params

`listSubscriptionDevices` requires `{ subscription_id, limit, offset, next?, prev? }` — not just `subscription_id`. The GET `/[id]/devices` route uses `parse_pagination(event.url.searchParams)` and passes all pagination fields through.

### LinkDevice FK error → 404

The original `LinkDevice.ts` service only caught UNIQUE constraint violations (→ 409). SQLite FK violations (unknown `device_id` or `subscription_id`) were being wrapped as 500. Added FK error detection in the catch block: messages containing `FOREIGN KEY` or `foreign key` → `validation.unknown_device_or_subscription` with status 404.

### `ListSubscriptionsRequest` type requires `include_deleted`

`z.optional().default(false)` in Zod gives a required output type (`boolean`, not `boolean | undefined`). Test code calling `sub_svc.listSubscriptions({ limit, offset })` without `include_deleted` fails TypeScript check. Must pass `include_deleted: false` explicitly.

### Smoke limitations

Full smoke (create + link device + toggle + soft-delete) requires a real registered source. Since no source modules are registered in production (007/008 not yet implemented), the create call correctly returns 400 `validation.unknown_source`. The smoke confirmed: list (200 empty), get nonexistent (404), create unknown source (400 with correct code). That is sufficient for this slice.

### Auth gate not testable via direct handler invocation

The auth gate lives in `hooks.server.ts` and is applied by the SvelteKit request pipeline. Calling route handlers directly (as done in tests) bypasses hooks. So the "Unauthenticated → 401" test is marked `[-]` (skipped — covered by slice 003's auth gate tests).
